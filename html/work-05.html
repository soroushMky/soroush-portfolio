01 Project Goals

Build a relational database for IPL matches, deliveries, players, and teams — then answer real cricket analytics questions: who are the top run-scorers in a tournament, which bowlers have the best economy rate, and how do two specific teams stack up head-to-head.

The dashboard itself was the harder constraint: it needed to look and feel like a real cricket scorecard, with interactive filters and team-level insights, not just a generic BI report.

02 Data & Tables

CSV/XLSX files — matches, deliveries, players, teams, and aggregated stats — loaded into a five-table PostgreSQL schema. matches holds match-level data (season, date, city, teams, winner, margin, venue, umpires). deliveries holds ball-by-ball data — match ID, inning, over, ball, batsman, bowler, runs, and dismissal info. players, most_runs, and team_home_away round out player metadata and pre-aggregated performance views.

-- matches: match_id, season, date, city, team1, team2, -- winner, win_by_runs, win_by_wickets, venue -- deliveries: match_id, inning, batting_team, bowling_team, -- over, ball, batsman, bowler, batsman_runs, -- extra_runs, total_runs, dismissal_kind -- players: player_name, dob, batting_hand, bowling_skill, country -- most_runs: batsman, totalruns, numberofballs, average, strikerate -- team_home_away: team, wins, matches, win_percentage
03 Key Analytics Queries

Top 5 run-scorers — a straightforward aggregation against the pre-computed most_runs table, ordered by total runs.

SELECT batsman, totalruns, average, strikerate FROM most_runs ORDER BY totalruns DESC LIMIT 5;

Best bowlers by economy rate — calculated directly from ball-by-ball deliveries data, with a minimum-overs filter so a bowler with one great over doesn't outrank someone with a real sample size.

SELECT bowler, SUM(total_runs) AS runs_conceded, COUNT(*) AS balls_bowled, ROUND(COUNT(*)::NUMERIC / 6, 2) AS overs_bowled, ROUND(SUM(total_runs)::NUMERIC / (COUNT(*)::NUMERIC / 6), 2) AS economy_rate FROM deliveries GROUP BY bowler HAVING COUNT(*) >= 24 -- 24 balls = 4 overs minimum ORDER BY economy_rate ASC LIMIT 5;

Head-to-head results — matches between any two specific teams, regardless of which one is listed as team1 or team2 in the raw data.

SELECT match_date, team1, team2, winner, win_by_runs, win_by_wickets, venue FROM matches WHERE (team1 = 'Team A' AND team2 = 'Team B') OR (team1 = 'Team B' AND team2 = 'Team A') ORDER BY match_date;
04 Tableau Dashboard Features

The Match Scorecard View mirrors a real scorecard — match header (season, teams, venue, date, result), innings summary (runs, wickets, overs, run rate), and batting/bowling tables styled to match.

A Selected Team parameter drives two linked views simultaneously: top 5 batsmen by total runs for that team, and best bowlers by economy rate when bowling for that team. Two more parameters — Team A and Team B — power a head-to-head sheet, using a calculated Boolean field that returns true for matches between those two teams regardless of listing order.

Every view updates instantly when the selected team or team pair changes — no separate filter clicks required across sheets.

05 Key Technical Decisions

Why PostgreSQL over a flat-file approach? Ball-by-ball delivery data multiplies fast — relational joins across matches, deliveries, and players let the analytical queries stay simple SQL instead of pre-flattening everything into a single bloated table.

Why a minimum-overs filter on economy rate? Without it, a bowler who threw four unhittable balls in a single over would outrank a genuinely consistent bowler — the filter (24 balls / 4 overs) is the difference between a real metric and a statistical artifact.

Why parameter-driven sheets instead of static views per team? 8–10 IPL teams means 8–10x the maintenance if each had its own sheet. One parameterized view scales to any team or matchup without duplicating dashboard logic.

06 What I Learned
01
Translating a sports analytics use case into a relational schema forces you to think about grain — match-level vs. ball-level data — before a single query gets written.
02
Writing SQL against questions a real fan would ask (top scorers, best economy, head-to-head) is a different discipline than writing SQL against abstract requirements.
03
A dashboard that visually mimics the thing it represents — a scorecard that looks like a scorecard — builds trust faster than a generic chart ever could.
07 Roadmap

Next: extend the schema across more seasons, add player profile pages, and layer in predictive models — win probability per match, player performance rankings — on top of the existing relational structure.
